Quem somos

Esta é uma API RESTful feita para auxiliar você a organizar sua vida financeira .

Nosso objetivo foi construir uma RESTful API que permitisse ao usuário - com seus dados pessoais: realizar cadastro, fazer login, detalhar seu perfil, editar seu perfil; - com as categorias de receitas e despesas: listar a categorias de receitas e despesas; - com suas transações: listar transações, detalhar transação específica, cadastrar transação, editar transação, remover transação, obter extrato de transações e, por fim, filtrar transações por categoria. Essas operações foram realizadas de modo seguro, por meio de senha criptografada e token de autenticação de cada usuário.

F.A.Q

Perguntas frequentes

A API RESTful é uma interface que dois sistemas de computador usam para trocar informações de forma segura pela internet. A maioria das aplicações de negócios precisa se comunicar com outras aplicações internas e de terceiros para executar várias tarefas. Uma interface de programação de aplicação (API) define as regras que você precisa seguir para se comunicar com outros sistemas de software. Os desenvolvedores expõem ou criam APIs para que outras aplicações possam se comunicar com suas aplicações programaticamente.

Este projeto diz respeito ao desafio do módulo 03, do curso de Desenvolvimento de Software com foco em Back-end, no Programa de Bolsas Potência Tech powered by iFood, proporcionado pela Cubos Academy. Neste módulo o desafio foi proposto para duplas de trabalho para que além das competências técnicas, pudéssemos desenvolver habilidades de trabalho em grupo.

Para realização deste projeto, utilizou-se a linguagem JavaScript, tendo NodeJS como ambiente de execução. Inicialmente, instalou-se o npm - gerenciador de pacotes do node. Depois instalou-se os pacotes express, pg(para conexão com o banco de dados Postgres), bcrypt (para a criação de hash no cadastro de senhas) e jsonwebtoken (para a criação de token de autenticação dos usuários). O pacote do nodemon foi instalado apenas como dependência de desenvolvimento.

A API RESTful, DinDin, teve como objetivo gerar um modelo de API de controle de finanças. Os instrumentos para sua criação foram o VSCode, o Postgres - Sistema de Gerenciamento de Banco de Dados -, e o Insomnia - para testar as requisições, os recursos e as respostas. As requisições foram feitas por meio das URI e eventuais parâmetros. A camada de rotas recebia as requisições e as encaminhava para a camada de controladores que, por sua vez, distribuiam as requisições aos respectivos serviços, onde deveriam ser executadas as regras de negócio. A API DinDin deveria

realizar cadastro, fazer login, detalhar seu perfil, editar seu perfil; - com as categorias de receitas e despesas: listar a categorias de receitas e despesas; - com suas transações: listar transações, detalhar transação específica, cadastrar transação, editar transação, remover transação, obter extrato de transações e, por fim, filtrar transações por categoria. Essas operações foram realizadas de modo seguro, por meio de senha criptografada e token de autenticação de cada usuário.

Para fazer a conexão dos endpoints/rotas com os controladores e com o banco de dados, foi necessário implantar uma camada de serviços. A camada de serviços ficou responsável por aplicar as regras de negócio (cadastrar, alterar, listar, excluir recursos) - sendo uma camada intermediária, ficando entre os repositórios e os controladores.

A conexão foi realizada com o SGBD - Postgres. Para tanto, criou-se uma pasta e um arquivo específico para realizar a função de conexão com o PG. Já para a realização das querys junto ao pg, foi criado um arquivo de repositório contendo todas as funções assíncronas necessárias à conexão com o banco de dados (para criar usuários, cadastrar senhas, cadastrar transações etc.)

Os controladores ficaram responsáveis por processar as requisições vindas dos endpoints, encaminhando-as para a camada de serviços, bem como responsáveis por responder às requisições recebidas. Quando necessário, os middlewares ou intermediários foram implementados para fazer validações indispensáveis ao funcionamento da API e a segurança no acesso aos dados.
